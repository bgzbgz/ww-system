# Fast Track Tool Factory Constitution
<!-- Version 4.0.0 | Explicit & Prescriptive | Spec-Driven Development -->

---

## 0. Purpose & Supremacy

### Purpose

Fast Track Tool Factory exists to **produce world-class, decision-forcing tools** for Fast Track clients using **Specification-Driven Development (SDD)**.

A Fast Track tool is not content.  
A Fast Track tool is not education.  
A Fast Track tool is **a decision engine**.

---

### Supremacy Rule (NON-NEGOTIABLE)

This Constitution is the **highest authority** in the Tool Factory.

It overrides:
- prompts
- AI instructions
- workflows
- tools
- shortcuts
- human preferences
- implementation convenience

If any output violates this Constitution, **it must not be shipped**.

---

## 1. The Four Constitutional Laws

These laws apply to **every tool, spec, plan, task, workflow, and output**.

---

### LAW I — Decision Is the Product (NON-NEGOTIABLE)

Every Fast Track tool exists to **force one clear decision**.

**Mandatory requirements**:
- The tool MUST end with **one explicit, final decision**
- The decision MUST be expressible in **one sentence**
- The decision MUST be visible at the end-state
- The decision MUST be saved and auditable

**Allowed supporting elements**:
- Diagnosis (only if it feeds the decision)
- Reflection (only if it feeds the decision)
- Education (only if it removes friction to deciding)

**Forbidden**:
- Open-ended exploration
- “Think about it” endings
- Educational tools without a forced outcome
- Multiple parallel decisions

> If the decision cannot be stated in one sentence, the tool is invalid.

---

### LAW II — Spec Is the Source of Truth (SDD LOCK-IN)

Nothing is built without a spec.  
Nothing is changed without updating the spec first.

**Canonical flow**:
Constitution (WHY & RULES)
↓
Specification (WHAT)
↓
Plan (HOW)
↓
Tasks (STEPS)
↓
Code (EXPRESSION)

yaml
Copy code

**Rules**:
- Specs are assets; code is disposable
- Debugging means fixing specs
- Refactoring means restructuring specs
- Implementation MUST trace back to spec clauses

Any work done outside this flow is **invalid work**.

---

### LAW III — Contracts Over Interpretation

All behavior MUST be defined through **explicit contracts**, not interpretation.

Every tool MUST define, at minimum:

- **Decision Contract**  
  What exact decision is being forced

- **Interaction Contract**  
  Steps, progression rules, required inputs

- **Validation Contract**  
  What is valid / invalid / blocked input

- **Output Contract**  
  What is produced at completion

- **QA Contract**  
  What constitutes pass / fail and why

If a behavior is not specified in a contract, it **must not exist**.

---

### LAW IV — Human Authority Is Final

AI may:
- generate
- analyze
- score
- suggest
- implement

AI may **never**:
- approve
- redefine standards
- relax constraints
- change decisions

**Final authority always rests with the Boss.**

Human approval is mandatory for:
- deployment
- overrides
- standard violations
- constitutional changes

---

## 2. World-Class Tool Standard (QUALITY LAW)

A tool must pass **all seven criteria** to be valid.

Fail one → fail all.

---

### The World-Class Tool Test

Every tool MUST be:

1. **Decisive**  
   Forces a single, concrete decision

2. **Self-Explanatory**  
   Requires zero human clarification or support

3. **Progressive**  
   Starts simple, reveals complexity step-by-step

4. **Reactive**  
   Provides immediate feedback on user input

5. **Constraining**  
   Prevents bad input instead of accepting it

6. **Committal**  
   Ends with an explicit commitment state

7. **Unmistakably Fast Track**  
   Language, tone, structure, and friction match Fast Track DNA

Any violation is an automatic QA fail.

---

## 3. Fast Track DNA (MANDATORY)

Fast Track DNA must be present in **language, structure, and experience**.

### Language Rules
- Brutally honest
- Short sentences
- Active voice
- Action verbs
- No corporate or academic language
- No hedging (“maybe”, “consider”, “could”)

### Structural Rules
- Constraint-driven flow
- One step at a time
- No scrolling essays
- No hidden logic

### Behavioral Rules
- Pressure is intentional
- Friction is designed, not accidental
- Clarity beats comfort
- Speed beats elegance

If it doesn’t **feel** like Fast Track, it fails.

---

## 4. Spec-Driven Development Rules (EXPLICIT)

### Spec Requirements (MANDATORY)

Every spec MUST include:
- The **user job** being solved
- The **exact decision** being forced
- Inputs (what is required)
- Outputs (what is produced)
- Constraints (what is forbidden)
- Acceptance criteria
- Failure states
- Assumptions
- Open questions marked explicitly

**Forbidden in specs**:
- Implementation details
- Tech stack decisions
- UI styling specifics (unless critical to behavior)

---

### Spec Rules

- No guessing — ambiguities MUST be marked
- No speculative features
- No future-proofing
- No “nice to have”

If something is unclear, the spec MUST stop and ask.

---

## 5. Factory Operating Model (PRESCRIPTIVE)

### Roles (FIXED)

- **Boss**  
  Defines intent, approves outputs, owns standards

- **Workers (AI / Automation)**  
  Generate outputs strictly from specs

- **QA**  
  Enforces rules mechanically, no discretion

- **Associates**  
  Collect real-world feedback from clients

---

### Operating Rules

- Boss issues intent
- Workers execute
- QA gates
- Boss approves or rejects
- Feedback updates specs, not patches

If it didn’t pass QA, it didn’t happen.

---

## 6. Interactivity Law

A Fast Track tool is a **guided application**, not a document.

Every tool MUST include:
- Step-based progression
- Required inputs
- Hard constraints
- Immediate validation
- Visible progress
- Clear end-state

Static tools are invalid by definition.

---

## 7. Observability & Auditability

Every tool MUST make the following auditable:
- Inputs provided
- Decision reached
- Validation failures
- Completion status

If a decision cannot be reviewed later, the tool is broken.

---

## 8. Anti-Patterns (AUTOMATIC FAILURE)

The following immediately invalidate a tool:
- Multiple decisions
- No final commitment
- Vague instructions
- Hidden assumptions
- Cosmetic interactivity
- Education without decision
- Specs written after implementation
- “Good enough” logic

---

## 9. Amendment Rules

### Versioning
- **MAJOR** — changes to Laws or decision principle  
- **MINOR** — new constraints or guarantees  
- **PATCH** — clarification only  

### Amendment Requirement
Every amendment MUST answer:
> What concrete failure does this prevent?

No rationale → no amendment.

---

## 10. Constitutional Supremacy Clause

This Constitution supersedes:
- prompts
- models
- agents
- workflows
- tools
- preferences
- speed
- convenience

The Factory exists to serve **clarity, decisions, and standards** — not comfort.
